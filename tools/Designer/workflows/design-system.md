# Workflow: Design System

## Purpose

Create a complete design system: color palette, typography scale, spacing tokens, component patterns, and a DESIGN.md that captures all rules in a portable, agent-readable format.

## When to Use

- Starting a new project and want to establish visual foundations
- Codifying an existing ad-hoc design into a system
- Phase 3+4 (Foundation + Compose) when the deliverable is the system itself

## Inputs

- Direction from Phase 2 (personality, aesthetic references, color mood, type voice)
- Domain context (web-apps, marketing, brand)
- Existing constraints (brand colors, required fonts, accessibility standards)

## Process

### Step 1: Generate Color Palette

**Route to:** `workflows/color-palette.md`

Execute the full color palette workflow. Output: CSS custom properties for primary scale, neutral scale, semantic colors, and surface colors.

### Step 2: Generate Typography System

**Route to:** `workflows/typography.md`

Execute the full typography workflow. Output: CSS custom properties for scale, weights, line heights, letter spacing, and font families.

### Step 3: Define Spacing Scale

Use an 8pt base unit with 4pt half-steps. This provides enough granularity without decision paralysis.

```css
:root {
  --space-0: 0;
  --space-0-5: 0.125rem;  /* 2px */
  --space-1: 0.25rem;     /* 4px */
  --space-2: 0.5rem;      /* 8px */
  --space-3: 0.75rem;     /* 12px */
  --space-4: 1rem;        /* 16px */
  --space-6: 1.5rem;      /* 24px */
  --space-8: 2rem;        /* 32px */
  --space-12: 3rem;       /* 48px */
  --space-16: 4rem;       /* 64px */
  --space-24: 6rem;       /* 96px */
  --space-32: 8rem;       /* 128px */
}
```

**Usage guidance:**
- `space-1` to `space-2`: Within components (padding, icon gaps)
- `space-3` to `space-4`: Between related elements (form fields, list items)
- `space-6` to `space-8`: Between sections within a view
- `space-12` to `space-16`: Between major page sections
- `space-24` to `space-32`: Page margins, hero spacing

### Step 4: Define Border and Shadow Tokens

**Border radius:**

| Token | Value | Use |
|-------|-------|-----|
| --radius-sm | 4px | Small elements (badges, chips) |
| --radius-md | 8px | Medium elements (cards, inputs) |
| --radius-lg | 12px | Large elements (modals, panels) |
| --radius-full | 9999px | Pills, avatars |

Adjust based on personality: geometric/professional uses smaller radii (4-8px); playful/friendly uses larger radii (12-16px). State the rationale.

**Shadows (elevation system):**

```css
:root {
  --shadow-sm: 0 1px 2px oklch(0% 0 0 / 0.05);
  --shadow-md: 0 4px 6px oklch(0% 0 0 / 0.07);
  --shadow-lg: 0 10px 15px oklch(0% 0 0 / 0.1);
  --shadow-xl: 0 20px 25px oklch(0% 0 0 / 0.1);
}
```

Tint shadows toward the primary hue for a more intentional feel: replace `0% 0 0` with `20% 0.01 [primary-hue]`.

### Step 5: Define Transition Tokens

```css
:root {
  --duration-fast: 100ms;    /* Hover, toggle */
  --duration-normal: 200ms;  /* State changes */
  --duration-slow: 300ms;    /* Panels, modals */
  --easing-default: cubic-bezier(0.4, 0, 0.2, 1);
  --easing-in: cubic-bezier(0.4, 0, 1, 1);
  --easing-out: cubic-bezier(0, 0, 0.2, 1);
}
```

No bounce or elastic easing (Prohibited Default).

### Step 6: Write DESIGN.md

Generate a DESIGN.md file that captures all design system rules in an agent-readable format. This file is the source of truth for the project's visual identity.

**Structure:**

```markdown
# Design System: [Project Name]

## Visual Direction
[Personality spectrum positions, aesthetic references, the "why" behind the visual choices]

## Color
[Color palette with roles, OKLCH values, hex fallbacks, contrast notes]

## Typography
[Font selection rationale, scale, weights, line heights, fluid sizing rules]

## Spacing
[Scale, usage guidance]

## Components
[Naming conventions, state patterns, shared behaviors]

## Rules
[Prohibited patterns, required patterns, accessibility minimums]
```

Each section should be dense and specific. Write rules the way you'd write them for another designer joining the project: enough to make correct decisions, not so much that they can't find what matters.

### Step 7: Generate CSS File

Combine all tokens into a single CSS file:

```css
/* [Project Name] Design Tokens */
/* Generated by Designer - see DESIGN.md for system rules */

:root {
  /* Color: Primary */
  /* ... */

  /* Color: Neutral */
  /* ... */

  /* Color: Semantic */
  /* ... */

  /* Color: Surfaces */
  /* ... */

  /* Typography */
  /* ... */

  /* Spacing */
  /* ... */

  /* Borders */
  /* ... */

  /* Shadows */
  /* ... */

  /* Transitions */
  /* ... */
}

/* Dark mode */
@media (prefers-color-scheme: dark) {
  :root {
    /* Override surface and text colors */
    /* ... */
  }
}
```

### Step 8: Tailwind Configuration (if applicable)

If the project uses Tailwind v4, output a `@theme` block:

```css
@theme {
  --color-primary-500: oklch(55% 0.22 250);
  /* ... all tokens mapped to Tailwind conventions ... */
}
```

For Tailwind v3, output a `tailwind.config.js` `extend` block.

## Quality Gates

- [ ] Color palette complete with all roles (primary, neutral, semantic, surface)
- [ ] Typography system complete with scale, fonts, weights, line heights
- [ ] Spacing scale defined with usage guidance
- [ ] Border radius and shadow tokens defined with rationale
- [ ] Transition tokens defined (no prohibited easing)
- [ ] DESIGN.md written and comprehensive
- [ ] CSS token file generated
- [ ] All tokens use three-layer architecture (primitive, semantic, component) where applicable
- [ ] Dark mode tokens included if the project needs them
- [ ] Every token has a clear purpose (no orphan values)
