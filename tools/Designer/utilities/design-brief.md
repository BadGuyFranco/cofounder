# Utility: Design Brief

## Purpose

Structured requirements gathering for design work. A good brief prevents the most common design failure: solving the wrong problem beautifully. Use this process when Phase 1 (Brief) needs more than the minimum three questions.

## When to Use

- Starting a significant design project (full page, design system, brand identity)
- When the user's request is vague or broad ("design my app")
- When multiple stakeholders have input
- Skip for small, specific requests ("design a button component for my existing system")

## Process

### Step 1: Core Questions

These four questions are required. Everything else can be inferred.

**1. What are we designing?**
Specific deliverable, not a vague goal. "A pricing page" not "improve the website."

If unclear, offer options:
> "Are we designing: (a) a single component, (b) a full page, (c) a design system, or (d) a set of pages/screens?"

**2. Who is the audience?**
The people who will see and use this design. What do they care about? What's their context?

| Dimension | Question |
|-----------|----------|
| Role | Who are they? (developer, executive, consumer, patient) |
| Expertise | How familiar are they with this product/domain? |
| Device | Where will they see this? (desktop, mobile, print, presentation) |
| Context | When and why are they looking at this? (working, shopping, researching, deciding) |

**3. What should it accomplish?**
The primary job of this design. One sentence.

| Purpose | Typical Design Response |
|---------|----------------------|
| Convert visitors to users | Landing page with clear CTA, social proof, feature highlights |
| Help users complete a task | Web app UI with efficient layout, clear navigation, feedback |
| Communicate brand identity | Brand system with personality, color, typography, application rules |
| Present information clearly | Content layout with hierarchy, readability, scannable structure |
| Impress or persuade | High-impact visual design with bold choices and polish |

**4. What constraints exist?**
Anything that limits design choices.

| Constraint Type | Examples |
|----------------|---------|
| Brand | Existing colors, fonts, logo, DESIGN.md |
| Technical | Must use Tailwind, must work in email, no JavaScript |
| Accessibility | WCAG AA required, specific audience needs |
| Content | Fixed content that can't change, required legal text |
| Competitive | Must differentiate from specific competitors |

### Step 2: Direction (if Phase 2 hasn't happened)

If the user hasn't established a visual direction, gather it now:

**Personality spectrum** (pick position on each):
- Playful ... Professional
- Bold ... Restrained
- Warm ... Cool
- Dense ... Spacious

**Reference products:** "Name 2-3 products or sites whose visual style is close to what you want. What specifically do you like about each?"

**What it should NOT look like:** Sometimes the clearest direction comes from knowing what to avoid. "Don't make it look like a fintech startup" is useful information.

### Step 3: Summarize and Confirm

Write a brief summary (5-8 lines) and confirm before proceeding:

> **Design Brief**
> **Deliverable:** [specific output]
> **Audience:** [who, context, device]
> **Purpose:** [primary job]
> **Constraints:** [brand, technical, accessibility, content]
> **Direction:** [personality, references, avoid]
>
> Ready to proceed with this direction?

If the user corrects anything, update and re-confirm. Don't proceed with unresolved ambiguity.

## Output

The design brief is not a deliverable; it's a working document used by Phases 2-5. Don't save it as a separate file unless the user asks. It informs the design process and is referenced when making decisions.

If the user asks for just a brief (without executing the design), output the summary format above as a standalone document.
